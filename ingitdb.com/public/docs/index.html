inGitDB Documentation

inGitDB turns a Git repository into a schema-validated, queryable database where every record is a plain YAML or JSON file. There is no server to run and no binary format to decode — the data lives in files you can read, edit, and diff with any text editor or git client.

Why inGitDB?

Git itself provides the history, branching, and merge story for free. If that sounds like the right trade-off for your project, read on — and consider contributing.

Quick Start

Get started with inGitDB in just a few commands:

Build the CLI

go build -o ingitdb ./cmd/ingitdb

Validate a database directory

# Validate current directory
ingitdb validate

# Validate a specific path
ingitdb validate --path=/path/to/your/db

# Fast CI mode - only changed records
ingitdb validate --from-commit=abc1234 --to-commit=def5678

List collections

# List all collections
ingitdb list collections

# List collections nested under a path
ingitdb list collections --in='countries/(ie|gb)'

# Filter by name
ingitdb list collections --filter-name='*city*'

CRUD Operations

# Create a record
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Read a record
ingitdb read record --id=geo/nations/ie

# Update a record
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Delete a record
ingitdb delete record --id=geo/nations/ie

Installation

inGitDB is written in Go and can be built from source:

# Clone the repository
git clone https://github.com/ingitdb/ingitdb-cli.git
cd ingitdb-cli

# Build the CLI
go build -o ingitdb ./cmd/ingitdb

# Optional: move to your PATH
sudo mv ingitdb /usr/local/bin/

CLI Commands

Complete reference of all available commands:

Command Status Description
version Implemented Print build version, commit hash, and date
validate Implemented Check every record against its collection schema
read record Implemented Read a single record by ID (local or GitHub)
create record Implemented Create a new record (local or GitHub)
update record Implemented Update fields of an existing record
delete record Implemented Delete a single record by ID
list collections Implemented List collection IDs (local or GitHub)
find Planned Search records by substring, regex, or exact value
query Planned Query and format records from a collection
materialize Planned Build materialized views into $views/
watch Planned Stream record change events to stdout
serve Planned Start MCP, HTTP API, or file-watcher server
--id format

The --id flag uses / as separator: <collection-path>/<record-key>

Example: --id=geo/nations/ie

Configuration

A minimal .ingitdb.yaml at the root of your database repository:

rootCollections:
  tasks: data/tasks/*   # each subdirectory becomes a collection
languages:
  - required: en

Learn more about configuration options in the configuration documentation.

GitHub Direct Access

inGitDB can read and write records in a remote GitHub repository without cloning it. The --github flag replaces --path and points the CLI at a GitHub repository over the REST API.

Public repositories (no token required)

# Read a record
ingitdb read record --github=owner/repo --id=countries/ie

# Pin to a specific branch, tag, or commit SHA
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# List all collections
ingitdb list collections --github=owner/repo

Private repositories

# Set the token once in your shell
export GITHUB_TOKEN=ghp_...

# Now you can read and write
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Each write operation creates a single commit in the remote repository. No local clone is required.

Features

What inGitDB Can Do Today

  • ✅ Schema validation with typed column definitions
  • ✅ CRUD operations on records (map[string]any collections)
  • ✅ Direct GitHub repository access (read and write)
  • ✅ Collection listing with regex filters
  • ✅ Fast incremental validation (commit-range scoped)
  • ✅ Version tracking with git history

What's Coming

  • ⏳ Full-text search with regex support
  • ⏳ Materialized views
  • ⏳ MCP server for AI agent integration
  • ⏳ File watcher for real-time change events
  • ⏳ HTTP API server
  • ⏳ Interactive merge conflict resolution

Architecture

The diagram below shows the write path: a CLI command loads the database definition, validates records, rebuilds materialized views, and leaves every change as a normal git commit.

YAML/JSON Files → Git Repository → Validation → Views Builder → Output (CLI/Go/AI)

Key Design Decisions

  • Plain files: Every record is a YAML or JSON file you can edit in any text editor
  • Git as storage: All version control, branching, and history comes from Git
  • Schema validation: Typed column definitions ensure data integrity
  • No server for reads: Reading data is just file system operations
  • Standard interfaces: DALgo integration for Go, MCP for AI agents

Learn more in the architecture documentation.

Go Library (DALgo)

pkg/dalgo2ingitdb implements the DALgo dal.DB interface, so any Go program can use inGitDB as a standard database abstraction.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Use inGitDB as a dal.DB instance
db := dalgo2ingitdb.NewDatabase(path)

// Standard DALgo operations
// ... your code here ...

This allows you to switch between different database backends (Firestore, SQL, inGitDB) without changing your application code.

Contributing

inGitDB is small enough that every contribution makes a visible difference. The best way to start is to point the CLI at a directory of YAML files and run ingitdb validate.

How to Contribute

  1. Read CONTRIBUTING.md for the pull-request workflow
  2. Browse open issues to find something to work on
  3. Open or comment on an issue before investing time in a large change

Bug reports, documentation improvements, and questions are all welcome.

Roadmap

inGitDB is being developed in nine delivery phases:

Phase 1: Schema Validator ✅

Schema validation, basic CRUD, collection listing

Phase 2: Query Engine ⏳

Search, filtering, and query operations

Phase 3: Views Builder ⏳

Materialized views and aggregations

Phase 4: Event Streaming ⏳

File watcher and change event streams

Phase 5: HTTP API ⏳

REST API server for remote access

Phase 6: MCP Server ⏳

Model Context Protocol for AI agents

Phases 7-9: Future

Migration tools, triggers, and GraphQL API

See the full roadmap for details.
